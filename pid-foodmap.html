Hệ Thống FoodMap Quận 4 — Cái Nhìn Tổng Quan

FoodMap Quận 4 là hệ thống du lịch ẩm thực số hóa tại Quận 4, TP.HCM. Hệ thống gồm 3 thành phần chính: Ứng dụng Mobile Android cho du khách khám phá ẩm thực trên bản đồ tương tác, Web Admin cho quản trị viên và chủ quán quản lý nội dung, và Backend API xử lý toàn bộ logic nghiệp vụ. Dưới đây là tổng quan các thành phần và chức năng cốt lõi.

📦 Ba thành phần hệ thống

📱

Mobile App (Android APK)

Ứng dụng Android bọc WebView chạy React. Du khách sử dụng không cần đăng nhập — xem bản đồ ẩm thực Leaflet, chọn Tour tham quan, GPS dẫn đường thời gian thực, nghe thuyết minh TTS đa ngôn ngữ (Việt/Anh/Trung), yêu thích quán, và du lịch ảo. Có cơ chế offline cache bằng LocalStorage.

🖥️

Web Admin (React + Vite)

Trang quản trị dành cho Admin và Seller. Admin quản lý toàn bộ: POI, Tour, Audio, Translation, Users, Duyệt yêu cầu, Dashboard Analytics. Seller đăng ký → nhận quán trống hoặc mở quán mới → cập nhật nội dung → chờ Admin duyệt. Gồm 15 trang React với phân quyền JWT.

⚙️

Backend & Database

2 server ASP.NET 8: Web API (port 6050) với 15 Controllers và 43 endpoints phục vụ Admin/Seller, Mobile API (port 6111) phục vụ Tourist. Database Supabase PostgreSQL với 14 bảng. Tích hợp FPT.AI TTS API (text-to-speech), Unified Proxy + Ngrok Tunnel.

🎯 Chức năng cốt lõi theo vai trò

👨‍💼

Admin — Quản trị viên

  • CRUD Điểm ẩm thực (POI) — tạo/sửa/xóa kèm ảnh, tọa độ GPS, danh mục, cascade delete
  • Quản lý Tour — tạo tuyến du lịch gắn nhiều POI bằng Npgsql, gợi ý Tour thông minh
  • Audio TTS — tạo thuyết minh bằng FPT.AI, CRUD quản lý file audio
  • Đa ngôn ngữ — CRUD bản dịch vi/en/zh cho mỗi POI
  • Duyệt yêu cầu — Approve/Reject 4 loại: Claim quán, Mở quán mới, Sửa thông tin (Stall/FoodPlace), Dịch thuật
  • Quản lý Users — tạo tài khoản, khóa/mở khóa, phân quyền
  • Dashboard Analytics — thống kê tổng quan, heatmap mật độ, visitor chart, active users realtime
🏪

Seller — Chủ quán ăn

  • Đăng ký tài khoản — tự tạo tài khoản Seller với BCrypt hash password
  • Nhận quán trống (Claim) — xem bản đồ quán vô chủ, bấm nhận → chờ Admin duyệt
  • Tạo quán mới — đề xuất mở quán kèm ảnh, vị trí, thông tin → chờ duyệt
  • Cập nhật Profile quán — sửa giá, mô tả, giờ mở cửa → tạo update request
  • Audio TTS riêng — tạo thuyết minh cho quán mình, file lưu server local
  • Dịch thuật riêng — đề xuất bản dịch Anh/Trung cho quán → chờ duyệt
📱

Tourist — Du khách (Mobile App)

  • Bản đồ ẩm thực — xem toàn bộ quán ăn trên Leaflet Map, popup chi tiết kèm ảnh
  • Chọn Tour & GPS — chọn tuyến tham quan, bật GPS dẫn đường theo Nearest-Neighbor
  • Thuyết minh TTS — khi đến gần quán (≤30m), tự phát audio giới thiệu đa ngôn ngữ
  • Chọn ngôn ngữ — chuyển đổi Việt / Anh / Trung cho toàn bộ nội dung
  • Yêu thích — toggle icon trái tim để đánh dấu quán ăn yêu thích
  • Du lịch ảo — trải nghiệm Tour không cần di chuyển thực tế
  • Offline Cache — lưu dữ liệu LocalStorage, sử dụng khi mất mạng

1. Chức năng Đăng nhập Hệ thống

Sequence Diagram
sequenceDiagram
    participant U as User Browser
    participant FE as React Frontend
    participant API as AuthController
    participant DB as Supabase DB

    U->>FE: Nhập username + password
    FE->>API: POST /api/Auth/login
    API->>DB: Tìm user theo username
    DB-->>API: User record hoặc null

    alt User không tồn tại
        API-->>FE: 401 Unauthorized - Sai tài khoản
        FE->>U: Hiện thông báo lỗi
    else User bị khoá Status = Locked
        API-->>FE: 401 Unauthorized - Tài khoản bị khoá
        FE->>U: Hiện thông báo tài khoản bị khoá
    else User tồn tại
        API->>API: BCrypt.Verify mật khẩu với hashPass
        alt Mật khẩu đúng
            API->>API: Tạo JWT token Claims: Name, Role, 2h expiry
            API-->>FE: 200 OK với token, role, username
            FE->>FE: Lưu token vào sessionStorage
            FE->>U: Redirect theo role Admin/Seller
        else Mật khẩu sai
            API-->>FE: 401 Unauthorized
            FE->>U: Hiện thông báo lỗi
        end
    end

3. Xử lý Xung đột và Chồng chéo POI gần nhau

Khi 2 hoặc nhiều quán ăn (POI) nằm san sát nhau, bán kính quét (activation_radius, mặc định 50m) của GPS có thể bao trùm nhiều quán cùng lúc. Hệ thống sử dụng thuật toán "Súng bắn tỉa GPS" — quét toàn bộ POI, tìm duy nhất 1 quán gần nhất bằng calculateDistance (Haversine), kết hợp cờ Dedup (currentShopId). Trường hợp đặc biệt: Nếu du khách đứng chính xác giữa 2 POI có khoảng cách bằng nhau, hệ thống ưu tiên POI có thứ tự xuất hiện trước (index nhỏ hơn) trong mảng allPlacesBackup từ database — do phép so sánh distMet < minDistance (strict less-than) chỉ cập nhật khi khoảng cách nhỏ hơn thật sự, giữ nguyên POI đầu tiên tìm được khi bằng nhau.

4. Xử lý Hàng đợi Đám đông (Zero-Lag Architecture)

Nếu hàng trăm, hàng ngàn du khách cùng đứng tại một điểm POI lúc này, xử lý hàng đợi (Queue) âm thanh trên Server sẽ dẫn đến sập hệ thống (bottleneck). Giải pháp của FoodMap là kiến trúc phân tán (Client-Side Computing) — để phần cứng điện thoại gánh tính toán thay vì Server.
